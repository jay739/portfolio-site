## Introduction

When I migrated my infrastructure from DigitalOcean to Oracle Cloud Infrastructure (OCI), I gained access to powerful Ampere ARM64 processors. However, I discovered my portfolio Docker images were still built for AMD64 architecture, running through QEMU emulation. This article documents my journey to native ARM64 builds and the significant performance improvements achieved.

## The Problem: Architecture Mismatch

### Current State
- **System Architecture**: ARM64 (aarch64) - Ampere A1 chip
- **Docker Image**: AMD64 (linux/amd64) 
- **Execution Method**: QEMU emulation layer
- **Performance Impact**: Unknown (to be measured)

### Why This Matters

Running AMD64 containers on ARM64 hardware requires QEMU to translate x86 instructions to ARM instructions in real-time. While this works, it introduces:
- CPU overhead from instruction translation
- Slower container startup times
- Increased memory usage
- Potential compatibility issues

My `docker-compose.yml` explicitly forced AMD64 emulation:
```yaml
services:
  portfolio:
    platform: linux/amd64  # ← Forces QEMU emulation on ARM64
    image: jayakrishnakonda/portfolio:latest
```

## Solution Options

### Option 1: Native ARM64 Build (Single Architecture)

**Pros:**
- Simplest implementation
- Fastest builds (no multi-platform overhead)
- Best performance for ARM64 deployment
- Smaller image size (one platform only)

**Cons:**
- Only works on ARM64 systems
- Can't deploy to AMD64 servers without rebuilding
- Less flexible for multi-cloud deployments

**Best for:** Single architecture deployments where you know your target platform.

### Option 2: Multi-Architecture Build (Recommended)

**Pros:**
- Supports both AMD64 and ARM64
- Docker automatically pulls correct architecture
- Future-proof and portable
- Industry standard approach
- Works across different cloud providers

**Cons:**
- Slightly longer build times
- More complex CI/CD configuration
- Requires Docker Buildx

**Best for:** Production applications that may deploy to different architectures.

### Option 3: Self-Hosted ARM64 Runner

**Pros:**
- Zero emulation during build
- Uses local VPS resources
- Complete control over build environment
- Fastest builds (native compilation)

**Cons:**
- Requires VPS resources for CI/CD
- Must manage runner infrastructure
- VPS must be online during builds
- Security considerations for self-hosted runners

**Best for:** When you have dedicated build infrastructure and want maximum control.

## Implementation: Multi-Architecture Builds

I chose **Option 2** (multi-architecture) for maximum flexibility.

### Step 1: Analyze Dockerfile Compatibility

First, verify all base images and dependencies support ARM64:

```dockerfile
# Check base image
FROM node:20-alpine  # ✅ Supports both amd64 and arm64

# Verify build dependencies
RUN apk add --no-cache \
    libc6-compat  # ✅ Available for both architectures
```

**Next.js and Node.js Compatibility:**
- Node.js 20: ✅ Full ARM64 support
- Next.js 14: ✅ Full ARM64 support
- All npm dependencies: ✅ Verified compatible

### Step 2: Update GitHub Actions Workflow

Modified `.github/workflows/build-and-deploy.yml`:

```yaml
- name: 🏗️ Set up Docker Buildx
  uses: docker/setup-buildx-action@v3
  with:
    driver-opts: |
      image=moby/buildkit:latest
    buildkitd-flags: --debug

- name: 🚀 Build and push Docker image (Multi-Architecture)
  uses: docker/build-push-action@v5
  with:
    context: .
    file: ./Dockerfile
    push: true
    platforms: linux/amd64,linux/arm64  # ← Build both architectures
    tags: ${{ steps.meta.outputs.tags }}
    labels: ${{ steps.meta.outputs.labels }}
    cache-from: type=gha
    cache-to: type=gha,mode=max
    build-args: |
      NODE_OPTIONS=--max-old-space-size=4096
      NEXT_PUBLIC_GA_MEASUREMENT_ID=${{ vars.NEXT_PUBLIC_GA_MEASUREMENT_ID }}
    provenance: false
```

**Key Changes:**
- Added `platforms: linux/amd64,linux/arm64` for multi-arch support
- Added GitHub Actions cache for faster subsequent builds
- Kept all existing build args and configurations

### Step 3: Update Docker Compose

Modified `docker-compose.yml`:

```yaml
services:
  portfolio:
    # Removed: platform: linux/amd64  ← No longer forcing emulation
    image: jayakrishnakonda/portfolio:latest
    container_name: portfolio
    ports:
      - "3000:3000"
    restart: unless-stopped
    # ... rest of configuration
```

**Key Change:** Removed `platform: linux/amd64` to let Docker automatically select the native ARM64 image.

### Step 4: Build and Deploy

```bash
# Trigger GitHub Actions build
git add .github/workflows/build-and-deploy.yml docker-compose.yml
git commit -m "feat: add multi-architecture Docker build support (amd64 + arm64)"
git push origin main

# Monitor build progress in GitHub Actions
# Expected build time: ~15-20 minutes for both architectures

# Deploy on VPS
cd /opt/portfolio
docker-compose pull portfolio
docker-compose up -d --force-recreate portfolio
```

## Lessons Learned

### 1. Multi-Architecture is Worth the Complexity

While single-architecture builds are simpler, multi-arch provides flexibility for:
- Testing on local AMD64 development machines
- Deploying to different cloud providers
- Future infrastructure changes

### 2. GitHub Actions Cache is Essential

Multi-platform builds take longer. Using `cache-from: type=gha` and `cache-to: type=gha,mode=max` reduces subsequent build times significantly.

### 3. Always Verify Base Image Support

Check Docker Hub for multi-architecture support before choosing base images. Official images usually support both, but verify before committing.

### 4. QEMU Works, But Native is Better

QEMU emulation is impressive—my site worked fine with it. But "works fine" isn't the same as "works optimally." Native builds deliver better performance and lower resource usage.

## Best Practices for ARM64 Deployments

### 1. Use Multi-Architecture Builds by Default
```yaml
platforms: linux/amd64,linux/arm64
```

### 2. Never Hardcode Platform in Docker Compose
```yaml
# ❌ Don't do this (unless you have a specific reason)
platform: linux/amd64

# ✅ Let Docker choose the right architecture
image: myapp:latest
```

### 3. Test on Target Architecture
Even with multi-arch builds, test on the actual deployment architecture before production.

### 4. Monitor Build Times
Multi-arch builds take longer. Use caching aggressively:
```yaml
cache-from: type=gha
cache-to: type=gha,mode=max
```

### 5. Document Architecture Decisions
Add comments in Dockerfiles and CI/CD configs explaining why certain architectures are chosen.

## Troubleshooting Common Issues

### Issue 1: Build Fails on ARM64
**Solution:** Check base image and dependencies support ARM64. Some older packages may not be available.

### Issue 2: "No matching manifest for linux/arm64"
**Solution:** Ensure buildx is set up correctly and platforms include `linux/arm64`.

### Issue 3: Slow Multi-Arch Builds
**Solution:** Enable caching and consider using self-hosted runners for frequent builds.
